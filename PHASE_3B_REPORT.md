## Fix B — Pro → Free downgrade flow

### Reasoning: immediate effect, not "at period end" — confirmed against the actual data model, not assumed

Checked what `initiatePayment` actually does (re-read `functions/src/index.ts`): it performs exactly one MTN `requesttopay` call per `_processPayment()` invocation — a one-time charge, not a recurring subscription created or tracked with MTN. Checked `Subscription`'s model and every place `startDate`/`endDate` are written (`mtnWebhook`, and previously `paymentWebhook`): both fields are **written** on every successful payment (now + 30 days) but I confirmed **nothing anywhere in the codebase reads `endDate` back** — no scheduled function checks it, no rule references it, no client code gates on it. There is no auto-expiry, no grace period, no "true up" job. `endDate` is inert, decorative data.

Given that, "downgrade at the end of the current paid period" isn't a real state this system tracks or could honor — there's no period to end *at*. **Decision: downgrade takes effect immediately**, exactly as the task anticipated as the likely-only-coherent option, now confirmed against the actual model rather than assumed.

### What happens to songs over the cap — checked, not guessed

Read `SongRepository.isAtSongLimit`: `choir.plan == ChoirPlan.free && choir.songCount >= 3` — this is the **only** place the cap is enforced client-side, and it only gates the "add a new song" action. Read the `songs` collection's Firestore read rule: `allow read: if isTenantMember(resource.data.choirId)` — no plan or count check at all, on read, anywhere (this is also true of the Fix 5 create-rule cap added in Phase 3, which only gates `create`, never `read`). And there is no client-side filtering of the song list by count either (checked `library_screen.dart`).

**Conclusion, verified via emulator test, not just code reading:** a downgrade never deletes, hides, or breaks access to existing songs — a choir with 7 songs that downgrades keeps all 7 fully visible and playable. The only behavioral change is that `isAtSongLimit` now evaluates `true` (since `plan == free && songCount(7) >= 3`), blocking new song creation until either the choir deletes down to under 3 or upgrades again. This is the standard "grandfather existing content, block new" pattern, and it's exactly what the rules layer already does mechanically — I didn't need to add anything to make this true, only confirm it. The confirmation dialog's copy reflects this precisely (see below), and differs depending on whether the choir is actually over the cap.

### What was built

**`cancelSubscription` Cloud Function** (`functions/src/index.ts`), same auth pattern as `initiatePayment`:
1. Verifies the caller's Firebase ID token (`admin.auth().verifyIdToken`) — real auth, not just "a header is present."
2. Server-side role check: reads `choir_memberships/{choirId}_{uid}` and requires `role in ['leader', 'director']` — **not** just any member, matching the task's requirement and mirroring `member_detail_screen.dart`'s own leader/director-only permission-management gate. A chorister gets 403.
3. Checks `subscriptions/{choirId}.status == 'active'` first — if there's nothing active to cancel, returns 409 rather than silently "succeeding" a no-op or double-cancelling.
4. Sets `subscriptions/{choirId}.status = 'cancelled'` and `choirs/{choirId}.plan = 'free'`, both via the Admin SDK (bypasses rules — same trust boundary as every other plan-changing write in this codebase). The client never writes either field directly.

**`billing_screen.dart`:**
- `_handleStepContinue()`'s step-2 branch now distinguishes a real downgrade (`selectedPlan == free && currentPlan == pro`) from "already free, nothing to do" — only the former triggers the new flow.
- **Confirmation dialog** (`_confirmAndProcessDowngrade`), shown before anything is called: reads the choir's actual `songCount` and picks between two copy variants — a plain "you'll lose Pro features" message if under the cap, or one that also explains the over-cap song-visibility behavior described above (with the real count filled in) if over it. "Keep Pro" / "Switch to Free" actions.
- On confirm: calls `cancelSubscription`, then — same discipline as Fix 4's payment polling — **watches `subscriptions/{choirId}` for the real resulting `status: cancelled`** rather than trusting the HTTP 200 alone (10-second timeout, since this is a synchronous server write rather than an async MTN callback; a `_PaymentOutcome.timeout` state covers a delayed/failed propagation instead of hanging the UI).
- `_ConfirmationStep` (the stepper's "Confirm" page, distinct from the dialog) no longer shows fake "UGX 40,000 / STK push" payment copy when the selection is actually a downgrade — it now branches to a "Switch to Free Plan / Amount: UGX 0 / no payment involved" summary instead, so the step immediately before the dialog doesn't contradict what the dialog is about to say.
- `_ProcessingStep` gained a `downgraded` outcome ("You're on the Free Plan" / "Your Pro subscription has been cancelled") and a `isDowngrade` flag so the in-flight spinner says "Switching to Free..." instead of "Processing payment... check your phone for the STK push" (which would be actively wrong for a downgrade — no MTN interaction happens at all).

**`subscription_repository.dart`:** added `cancelSubscription(choirId)`, same shape/error-handling pattern as `initiatePayment` (verified ID token, POST, plain-English error surfaced via `PaymentInitiationException`).

**`enums.dart`:** added `SubscriptionStatus.cancelled` — required so `'cancelled'` (written server-side) round-trips through `Subscription.fromJson`/`toJson` instead of falling through to the enum's default.
