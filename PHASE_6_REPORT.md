## Fix 2 — Android App Links: **found a real, previously-undiscovered break while tracing it**

**Manifest itself is correctly configured for the production project:** `AndroidManifest.xml`'s intent-filter (`autoVerify="true"`, `scheme="https"`, `host="kwayapro.app"`) targets a plausible real domain, not a placeholder, and doesn't reference the old `kwayapro-production` project-ID confusion resolved in Phase 3 — that confusion was about Cloud Functions URLs, unrelated to this domain. `google-services.json` and `firebase_options.dart` both confirm `kwayapro-app`/`com.kwayapro.kwayapro` consistently.

**But tracing the guest-director invite flow through to the actual link it generates — as instructed, not just eyeballing the manifest — found it doesn't use this domain at all.** `guest_director_screen.dart`'s `_generateAndShare()`/`_shareLink()` were building invite links as `https://kwayapro.page.link/rehearsal-invite/$token` — a **Firebase Dynamic Links** URL, an entirely different domain than the one declared in the App Links manifest. Checked Firebase's own deprecation notice (`firebase.google.com/support/dynamic-links-faq`): **Firebase Dynamic Links shut down completely on 2025-08-25** — every `*.page.link` URL has returned HTTP 404 since. Given today's date, this means every guest-director invite link generated by this screen has been **completely dead on arrival for nearly a year**, independent of anything about the App Links configuration — the App Links setup was never even the thing being exercised.

**Fixed:** both link-generation sites now build `https://kwayapro.app/rehearsal-invite/$token` — the app's own real App Links domain, which already correctly matches `app_router.dart`'s `/rehearsal-invite/:token` route and the manifest's declared host. This is a two-line, surgical change to link-string construction in the one screen responsible for generating this specific link — flagging clearly that I made a judgment call to fix rather than only report, since (a) it directly blocks Fix 2's own goal of confirming these links "actually resolve through this configuration correctly" — they couldn't, at all, via any configuration, while pointed at a dead service — and (b) the fix carries essentially zero risk (a URL string, not app logic).

**Still missing, not fixed in this phase (out of scope — requires infrastructure outside this repo, not build config):** no `assetlinks.json` exists anywhere in this repo, and there's no Firebase Hosting (or other web hosting) configuration for `kwayapro.app` in `firebase.json` at all. Android App Links verification **cannot succeed** without `https://kwayapro.app/.well-known/assetlinks.json` being served with the correct SHA-256 app signing certificate fingerprint (confirmed against `developer.android.com/training/app-links` — no version-sensitive process changes since this was set up, beyond an unrelated Android 15 addition for dynamic runtime link updates that doesn't affect this setup). **This needs your action**: host `assetlinks.json` at that exact path (Firebase Hosting is the natural fit given the rest of the stack, but isn't configured in this repo yet) and verify via Play Console / a real device after the next release — App Links verification cannot be checked from this environment.

Also still true, unchanged from the original audit and out of scope for "Android App Links": iOS has no Associated Domains / Universal Links entitlement at all, separate from the placeholder iOS Firebase `appId` already tracked as a known gap.

## Fix 5 — pubspec.yaml version pinning: **reviewed, nothing needed changing**

Checked every dependency (prod and dev) for `any`, unbounded, or suspiciously wide constraints. **None found** — every single entry already uses a standard caret (`^`) range. The judgment call worth stating explicitly: Dart's caret operator already treats pre-1.0 packages (`audio_session: ^0.1.18`, `intl: ^0.19.0`, `rxdart: ^0.28.0`) as bounded to their *minor* version rather than major (`^0.1.18` means `>=0.1.18 <0.2.0`), so these aren't "loosely constrained" in the syntax sense — the real risk with 0.x packages is that maintainers aren't strictly bound by semver even within that already-narrow range, which is a package-trust judgment, not something tightening the pubspec constraint further would meaningfully change. I didn't find any dependency where the currently-declared caret floor was meaningfully looser than what's actually needed, so I made no changes — tightening working, already-reasonable constraints for no functional reason isn't warranted.

The three explicit `dev_dependencies` added in Phase 5b are already sensibly pinned, matching their currently-resolved versions exactly: `path_provider_platform_interface: ^2.1.2` (resolved 2.1.2), `connectivity_plus_platform_interface: ^2.1.0` (resolved 2.1.0), `plugin_platform_interface: ^2.1.8` (resolved 2.1.8) — floor matches resolved version, nothing to tighten.
